Security & Encryption

Kheyr is designed so your carrier SMS can stay on your phone by default, with optional cloud sync that keeps message bodies encrypted end-to-end from the server’s perspective. This page describes the cryptography and key handling implemented in the published app.

Security model at a glance

Local at rest

Room database encrypted with SQLCipher. Passphrase stored in Android EncryptedSharedPreferences backed by AES-256-GCM master keys.

Cloud sync

SMS bodies encrypted with AES-256-GCM on Android before upload. Servers store ciphertext and minimized metadata only.

Desktop access

Sync keys delivered during approved QR pairing. Revoked devices lose decryption and API access immediately.

1. Local storage encryption

All SMS threads and messages stored in Kheyr’s on-device database are protected at rest.

Technology

  • SQLCipher — full-database encryption for the local Room database file (kheyr_sms.db).
  • Android Keystore — the SQLCipher passphrase is generated and stored using encrypted shared preferences with AES-256-SIV key encryption and AES-256-GCM value encryption.

Protected fields

Sensitive columns including message body, sender/receiver address, and display names are stored inside the encrypted database. Search and spam scoring operate on decrypted local data only on your device.

Uninstalling the app or clearing app data removes local keys and message storage unless you have a separate backup you created through export features.

2. Sync encryption (client-side)

When you opt in to sync, Kheyr encrypts SMS content on your Android device before any upload. Our backend never receives plaintext SMS bodies and cannot decrypt synced message content.

Algorithm

ComponentImplementation
Message body cipherAES/GCM/NoPadding (AES-256-GCM)
Nonce12-byte random nonce per message, transmitted alongside ciphertext
Authentication tag128-bit GCM tag
EncodingBase64-encoded nonce and ciphertext in sync payloads
Sync key storageAndroid Keystore alias kheyr_sync_key

Key lifecycle

  1. A per-account sync encryption key is generated on your Android device when sync is set up.
  2. The key is stored in Android Keystore and is required before any encrypted upload proceeds.
  3. SMS bodies are encrypted with this key immediately before being placed on the sync upload queue.
  4. Only ciphertext, algorithm identifier, nonce, and non-sensitive metadata are transmitted to the server.
  5. Desktop clients receive key material during an approved QR pairing exchange and decrypt locally.
  6. Revoking a desktop device or deleting cloud data removes server-side access to future sync payloads for that device.

Field protection policy

Sync uploads apply field-level protection rules:

  • Encrypted — message body, sender, recipient, address fields
  • Salted hash — phone numbers used for discovery or correlation where full numbers are not required
  • Plain metadata — timestamps, direction, message type, spam flags, pin/archive state, and other operational fields needed for sync without revealing content

3. Transport security

  • All API and sync traffic uses HTTPS (TLS).
  • Realtime desktop relay and sync updates use authenticated WebSocket connections (SignalR) with access tokens.
  • Desktop SMS relay requests carry encrypted target numbers and message bodies when submitted from desktop clients.

4. Desktop pairing

Desktop apps cannot read your synced SMS until you explicitly approve pairing on your phone.

  • Pairing sessions are time-limited QR codes.
  • Android shows the requesting device name and platform before approval.
  • Approved desktops register a device public key and receive credentials needed to decrypt synced data.
  • Revocation from Android immediately blocks API and realtime access for that desktop.
  • Desktop apps store decrypted data in local encrypted storage on each platform.

5. Spam processing privacy

Spam rules are downloaded as signed/versioned rule sets. Scoring executes entirely on-device. User spam feedback uploads hashed identifiers and rule metadata—not raw SMS bodies—so global rules can improve without exposing message content.

6. Direct messages

Internet-based direct text and picture messages are a separate channel from carrier SMS. In the current release, direct messages are not end-to-end encrypted; they are protected in transit with TLS and stored on Kheyr servers for delivery and multi-device access. We clearly label direct messages in the UI so you can distinguish them from SMS.

8. Your controls

  • Keep sync disabled to avoid any cloud upload.
  • Delete cloud data or your entire account from Settings → Privacy & Security.
  • Revoke individual desktop devices at any time.
  • Rotate or remove access by signing out and deleting sync keys as part of account deletion.

9. Reporting security issues

If you discover a vulnerability, email support@kheyr.app with steps to reproduce. We appreciate responsible disclosure and will respond as quickly as possible.
